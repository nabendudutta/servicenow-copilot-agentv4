## Description

ArgoCD on the production Kubernetes cluster shows 15 applications in 'OutOfSync' status following a security team RBAC policy change that restricted the ArgoCD service account permissions. ArgoCD cannot reconcile application state and deployments are blocked.

## Resolution Notes

ROOT CAUSE ANALYSIS:
The security team applied a new ClusterRole policy that removed the 'create' and 'patch' verbs from the ArgoCD application controller service account for the core/v1 'secrets' resource. This was part of a least-privilege hardening initiative but was applied without consulting the ArgoCD platform team.

RESOLUTION STEPS TAKEN:
1. Reviewed ArgoCD application controller logs:
   kubectl logs -n argocd deployment/argocd-application-controller | grep 'error'.
2. Identified 'secrets is forbidden: User cannot patch resource secrets' errors    indicating RBAC permission denial.
3. Retrieved the current ArgoCD ClusterRole:
   kubectl get clusterrole argocd-application-controller -o yaml.
4. Compared against the required permissions from the official ArgoCD RBAC docs.
5. Coordinated with the security team to review the specific permissions required    and agreed on the minimum needed set.
6. Patched the ClusterRole to restore 'create' and 'patch' on 'secrets' with a    resource name restriction limiting scope to ArgoCD-managed namespaces only.
7. Applied the updated RBAC manifest:
   kubectl apply -f argocd-clusterrole-updated.yaml.
8. Triggered a manual refresh on all 15 OutOfSync applications from the ArgoCD UI.
9. Confirmed all applications transitioned to 'Synced' and 'Healthy' status.
10. Documented the ArgoCD required RBAC permissions in the internal security     baseline document and added ArgoCD as a stakeholder in future RBAC change     review meetings.

PREVENTIVE MEASURES:
Added ArgoCD RBAC validation as a step in the Kubernetes security hardening change approval workflow.
