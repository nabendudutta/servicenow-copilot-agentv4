## Description

Ansible Tower job templates targeting RHEL 8 hosts are failing with 'Permission denied (publickey)' SSH errors since 08:00 UTC today. Automated patching and configuration management workflows are blocked for 47 servers.

## Resolution Notes

ROOT CAUSE ANALYSIS:
The SSH private key stored in the Ansible Tower credential object expired and was rotated by the Secrets Management team (CyberArk) as part of a scheduled 90-day key rotation policy. The Tower credential was not updated with the new key.

RESOLUTION STEPS TAKEN:
1. Reviewed Tower job output logs to confirm the exact error:    'Permission denied (publickey,gssapi-keyex,gssapi-with-mic)'.
2. Tested manual SSH from Tower host using the stored key:
   ssh -i /tmp/test_key ansible-svc@rhel-host-01 â€” confirmed key rejection.
3. Contacted the Secrets Management team to retrieve the newly rotated SSH private    key for the 'ansible-svc' service account from CyberArk vault.
4. Logged into Ansible Tower UI â†’ Credentials â†’ 'RHEL-Prod-SSH-Cred'.
5. Updated the SSH Private Key field with the new key provided by Secrets team.
6. Saved the credential and re-launched a test job template against one host.
7. Confirmed successful SSH authentication and task execution on the test host.
8. Re-launched all failed job templates; all 47 hosts completed successfully.
9. Documented the key rotation schedule in the runbook and set a calendar reminder    for 85 days hence to proactively update the Tower credential before expiry.
10. Raised a request to integrate Ansible Tower with CyberArk API for automated     credential sync on rotation events to prevent recurrence.

PREVENTIVE MEASURES:
CyberArk-to-Tower dynamic credential sync integration scheduled for Q3.
